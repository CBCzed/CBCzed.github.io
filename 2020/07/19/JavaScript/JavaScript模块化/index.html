JavaScript模块化

发表于 2020-07-19 阅读次数： 阅读次数： Valine：

导读

　　随着 ES6 的发展，模块化在项目中的运用越来越多，接触过模块化之后，相信大家都能感受到模块化给我们带来的优势，避免命名冲突、解耦代码、高复用性、高可维护性等；在 ES6 模块出现之前，社区中就有一些比较优秀的模块化的规范，如 CommonJS 、 AMD 、 CMD 等，甚至在更早时，为了使代码更佳的规范，也使用自执行函数 IIFE 或命名空间来规范代码，然而有些时候，我们只是看着用，但并没有去真正了解下如何用好模块化，本文将主要介绍 CommonJS 、 ES6 规范。

CommonJS

相信用过 Node 的小伙伴们，对 CommonJS 一定不会陌生， Node 就是遵循 CommonJS 规范来开发的。但随着 ES6 模块化的实现， Node 在 v13.2 版本开始，默认打开对 ES6 的方式支持，只不过文件名的格式需要使用 .mjs 后缀。
CommonJS 规范认为，每个文件就是一个模块，有自己单独的作用域；在一个文件里面定义的变量、函数、类，都是私有的，对其他文件不可见；每个模块内部， module 变量代表当前模块，这个变量是一个对象，它的 exports 属性（即 module.exports ）是对外唯一的接口。使用 require 命令加载某个模块时，其实是加载该模块的 module.exports 属性。

require命令

require 命令用于引入模块、 JSON、或本地文件，然后返回该模块的 exports 对象，使用方法为， require(id) ， id 为模块名称路径，其加载规则为，当 id 为模块名称时，则相应的会从 node_modules 目录下加载模块，若是路径，则会按照路径来引入模块或文件；
require 命令并不是全局的，当调用时，其指向的是 module.require ，而 module.require 指向的是 Node 的内部命令 Module._load ；
在 Node 中， require 还包含以下属性：

  • require.cache 是一个对象，所有被引入的模块都将缓存在这个数组里，对象的 key 为模块的路径， value 为 module 对象，若需要清除缓存，则要删除该对象；
  • require.resolve(request[, options])是一个方法， path 表示需要解析的模块路径，可选参数 options 对象包含 paths 属性， paths 数组表示解析模块位置的路径，若存在值，则会覆盖默认的解析路径；该方法最终返回解析后的文件名，但不会加载该模块；
  • require.main 是一个 module 对象，表示当 Node 进程启动时加载的入口脚本
  • require.extensions 是一个扩展对象，表示如何处理这些文件的扩展名，该属性已经废弃（但是打印 require 时仍旧可以看到该属性）

特点

CommonJS 规范有以下几个特点：

1.所有代码都运行在模块作用域，不会污染全局作用域；
这一点相信大家都能理解，这也是模块的规范，模块的代码都是私有的，且只有通过 module.exports 暴露接口才可以被访问，并且 this 对象指向该模块；

2.模块可以多次加载，但是只会在第一次加载时运行一次，然后运行结果就被缓存了，以后再加载，就直接读取缓存结果。要想让模块再次运行，必须清除缓存；

3.模块加载为同步加载方式，这也就是 CommonJS 不适用于浏览器端的主要原因；
4.模块加载的顺序，按照其在代码中出现的顺序
后面这两点就比较好理解了，模块的是同步加载，如是在浏览器端使用，则会因为等待时间过长而导致用户体验不好，而在服务端，数据都是存储在硬盘中，所以加载速度就比较快；当然，社区中有人开发了 browserify 库，使得浏览器端使用 CommonJS 变成了可能。

加载机制

require 命令引入的是模块的 exports 对象，在 CommonJS 中，模块的加载机制是，输入的是被输出的值的拷贝；也就是说，一旦输出一个值，模块内部的变化就影响不到这个值；

ES6 module

ES6 模块化的规范实现很简单，就是两个 API ， import 导入及 export 导出；由于 ES6 的模块默认采用严格模式，所以在开发模块时要注意严格模式的限制，严格模式的限制如下：

  • 变量必须声明后再使用
  • 函数的参数不能有同名属性，否则报错
  • 不能使用 with 语句
  • 不能对只读属性赋值，否则报错
  • 不能使用前缀 0 表示八进制数，否则报错
  • 不能删除不可删除的属性，否则报错
  • 不能删除变量 delete prop ，会报错，只能删除属性 delete global[prop]
  • eval 不会在它的外层作用域引入变量
  • eval 和 arguments 不能被重新赋值
  • arguments 不会自动反映函数参数的变化
  • 不能使用 arguments.callee
  • 不能使用 arguments.caller
  • 禁止 this 指向全局对象
  • 不能使用 arguments.callee 和 fn.arguments 获取函数调用的堆栈
  • 增加了保留字（比如 protected 、 static 和 interface ）

ES6 模块的设计思想是尽量的静态化，使得编译时就能确定模块的依赖关系，以及输入和输出的变量； CommonJS 只能在运行时确定这些东西。比如 CommonJS 模块就是对象，输入时必须查找对象属性；而 ES6 模块不是对象，而是通过 export 命令显式指定导出的代码，再通过 import 命令导入；由于是编译时加载，不仅效率比 CommonJS 高，也使得静态分析成为了可能，但由于目前个浏览器对 ES6 模块的支持度还不是很友好，所以还需要通过 Babel 等转译工具进行编译。

以上就是简单的 ES6 模块化例子，导出的方式有两种，个人比较建议使用方式二，这样可以更佳清晰的查看导出的数据。

export命令

export 命令用于规定模块暴露的接口；外部文件只能通过暴露的接口来访问模块的数据；
使用 export 导出接口时，也可以通过 as 关键只为接口重命名；

export 导出的接口，与其对应的值是动态绑定关系，即通过该接口，可以取到模块内部实时的值，这一点与 CommonJS 有很大的区别。
export 语句可以在文件顶层任意使用，但不能在其它块语句内使用；

import命令

import 命令用于导入模块，且只会加载所导入的方法，其余模块的方法不会加载；且如果从同一模块中导入多次，最终也只会执行一次。
与 export 一样，用 import 时，也可以使用 as 关键字为导入的接口重命名；

import 命令支持整体加载，而不用一个一个的导入；

import 导入的变量都是只读的，所以不可以对变量本身进行操作，但若导入的变量是对象类型的，则可以对其属性进行操作；

动态加载 import() 函数

import 命令是在编译时运行，且只能在顶层声明，这就限制了动态加载模块的功能，所以 ES2020 引入了 import() 函数，用于动态加载模块，且无需在顶层也可以运用；
import() 函数为异步加载，返回一个 Promise 对象；

import(`./xxx.js`)
  .then(module => {
    // module 为模块文件返回的结果
  })
  .catch(err => {

  })

import() 函数常用于按需加载、条件加载、动态的模块路径等，像 Vue 、 React 路由的按需加载就是使用该函数实现的。

加载机制

ES6 模块加载与 CommonJS 有着本质的不同， ES6 import加载的是对模块的动态引用，所以当模块内部发生变化时，都可以获取到变量的最新数据；这也是 ES6 模块的循环加载的解决方式；只要在引用模块时确保变量已经定义了即可。

小结

ES6 模块与 CommonJS 模块存在本质上的不同，主要有以下两个大的差异：

  • CommonJS 模块输出的是一个值的拷贝， ES6 模块输出的是值的引用。
  • CommonJS 模块是运行时加载， ES6 模块是编译时输出接口
